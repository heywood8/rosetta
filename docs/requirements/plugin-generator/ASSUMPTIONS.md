# plugin-generator — Assumptions, Quirks, Open Questions

## Assumptions (AC)

- **AC-1 — Uniform generation is the intent; two-pass derivation is an accident (now codified).** The original code generates the four "main" targets first, then derives the two standalone targets by reading from the already-generated main plugin trees (`scripts/plugin_generator.py:901` reads `plugins_root / spec.source_plugin`; `:1152` second loop). Per the maintainer, this coupling caused repeated defects and is *not* a requirement. Target state — now a requirement, not just an assumption: all six targets are generated independently from the instruction source by the same pipeline (FR-CLI-0040; standalones explicitly uniform-from-source in FR-VAR-0050/0051). Requirements capture per-variant **output properties**, never the derive-from-main mechanism. This also retires the `.cursor/.cursor/` self-nesting guard (QF-4).
- **AC-2 — Parity bar is byte-for-byte.** A re-implementation must reproduce the current generator's output **byte-for-byte** for equivalent inputs — verified by an empty recursive diff (NFR-0001). The generator controls its own serialization (JSON/TOML/Markdown) to match exact bytes; declared-new behavior (e.g. `--domain`, dry-run) has no baseline and is excluded. (Supersedes the earlier functional-only stance.)
- **AC-3 — Hook bundles are an external input.** `src/hooks/dist/bundles/<target>` and `src/hooks/dist/shell/` are produced by a separate build step. The generator assumes they exist when a release requires deterministic hooks. This covers only the compiled `*.js` runtime bundles, not the preserved manifests/templates (see AC-3a).
- **AC-3a — Preserved manifests and templates are a committed source, not an output-tree precondition (RESOLVED).** Earlier behavior depended on each target's IDE manifest (`plugin.json`), hook templates (`*.tmpl`), config-folder contents, and any `.mcp.json` already existing inside the committed output tree (`plugins/<target>/...`), surviving the wipe as preserved paths — so generating into a clean/empty output directory required manually seeding them (observed in discovery §7 and the validation report). Resolved: these preserved files now have a committed source under `src/rosettify-plugins/plugins/<target>/` (DATA-CFG-0005) that `pluginCopy()` copies into the output before generation (FR-SEED-0001, FR-ARCH-0053). Standalone preserved files derive from the parent target's source set (FR-SEED-0002). There is no longer any "survive the wipe" semantics: `pluginCleanup()` empties the output and `pluginCopy()` re-establishes the preserved files from source every run.
- **AC-4 — Instruction source layout is stable.** Source content lives under typed folders (`agents/`, `rules/`, `skills/`, `workflows/`, `configure/`, `templates/`). Agent files are `agents/<name>.md` exactly one level deep (`scripts/plugin_generator.py:840`).
- **AC-5 — Run via npx.** The re-implementation targets a TypeScript/Node runtime invoked through `npx`, replacing the Python entry point; the Handlebars dialect and filesystem operations have direct Node equivalents.
- **AC-6 — Logical model keys are release-neutral.** Frontmatter `model:` values are logical keys (`opus`, `sonnet`, `gpt-5.4`, …); the per-IDE version strings they map to are config expected to change.

- **AC-14 — Copilot produces three distinct hooks.json files (CONFIRMED INTENDED).** The `core-copilot` target emits `hooks.json` (root), `.github/plugin/hooks.json`, and `hooks/hooks.json` — three separate files at three distinct paths. The root and `.github/plugin/` copies are byte-identical (alternate-name duplication, FR-VAR-0031, FR-COPY-0033); `hooks/hooks.json` is the standalone-form rendered output with distinct content (`"sessionStart": []`). This is confirmed by MD5 comparison in r2/r3 baselines and is codified in FR-VAR-0030/0031 and STRUCTURES.md. The original STRUCTURES.md diagram omitted `hooks/hooks.json` and mischaracterized the root copy as a `fileRename()`.

## Flagged Quirks / Findings (QF) — current behavior, not necessarily intended

- **QF-1 — Order-sensitive bootstrap prefix (RESOLVED).** The bootstrap prefix attaches to the *first bootstrap-classified file found* per target; reordering the bootstrap file list silently moves the prefix (`scripts/plugin_generator.py:501-507`, `:599`). Resolved in the target state: the bootstrap-file order is an explicit, required, deterministic manifest (FR-HOOK-0009) and the prefixed document is an explicitly designated lead (FR-HOOK-0003) — not position-dependent. Note: bootstrap-file order remains **significant** (it fixes payload entry sequence and prefix placement); it is now contractual rather than accidental.
- **QF-2 — Oversize payloads are a soft error (CONFIRMED INTENDED).** A bootstrap context entry exceeding 10000 characters is reported, still emitted, and sets a non-zero exit status. This soft-error behavior is intended (not a defect): the run completes so all problems surface together, and the non-zero exit signals the violation. Codified in NFR-0004.
- **QF-3 — Copilot duplicate-hook lock is a workaround.** Copilot fires session hooks twice, so per-entry file locks are generated to dedupe (`scripts/plugin_generator.py:469-498`, `:614-638`). This is a workaround for an IDE defect; the underlying need is "each bootstrap entry takes effect once per session." Other IDEs rely on native dedup (`once`, built-in).
- **QF-4 — Standalone subfolder self-nesting guard (RESOLVED).** When a source folder name equals the standalone's target subfolder (e.g. `.cursor`), special-case merging avoids `.cursor/.cursor/` (`scripts/plugin_generator.py:914-919`). This is a defensive patch arising from the derive-from-main approach (AC-1). Resolved: a uniform generator writing directly to the target layout from the instruction source never produces the nesting (FR-VAR-0050), so the guard is unneeded.
- **QF-5 — Silent skips (RESOLVED).** Missing bootstrap-file variants and missing templates were skipped silently or with only a stderr warning (`scripts/plugin_generator.py:589`, `:685`). Resolved: every skip is logged — absent bootstrap variants (FR-HOOK-0001), missing templates (FR-GEN-0010), and per-processor decisions (FR-ARCH-0050).
- **QF-6 — Mixed concerns in one module (RESOLVED in target state).** Filesystem mechanics, IDE-specific string escaping (bash/PowerShell/TOML/JSON), model maps, and orchestration live in one file, and `copy_core_tree` fuses path renaming with content rewriting. This is the "awful architecture" the maintainer wants split. Resolved by the unified processor model: single-responsibility processors (NFR-0007, FR-ARCH catalog), with `fileRename()` path-only and `pluginRewriteReferences()` content-only.
- **QF-7 — Shallow regex corrupted prose (RESOLVED, regression lesson).** A prior re-implementation attempt rewrote the bare token `agents`→`.codex/agents` with an unanchored/substring match, corrupting the ordinary word "agents" throughout every document — even though that responsibility was paths only. Resolved and generalized: **every** regular expression and match in the generator must be strict and anchored/bounded — never shallow substring or bare-word — and reference rewriting consumes concrete resolved path pairs at token boundaries, not loose regex (FR-ARCH-0037, FR-ARCH-0049). The same lesson fixes the tag-membership substring risk (FR-GEN-0003).

## Target-state assumptions (added from rewrite design)

- **AC-7 — Unified architecture: FR-ARCH is normative; as-is FRs recast in place.** `FR-ARCH.md` defines the single target architecture — an immutable VFS (with structural sharing) and a **two-tier** pure single-responsibility processor pipeline (`FileProcessor` over a `FileProcessingFrame`, `PluginProcessor` over a `PluginProcessingFrame`). The reverse-engineered behavior FRs (`FR-COPY`, `FR-GEN`, `FR-HOOK`, `FR-VAR`) have been **recast in place** onto this model — not dropped: each now states its behavior as a `FileProcessor`/`PluginProcessor`/`SpecEntry` (e.g. rename→`fileRename()` path-only, content rewrite→`pluginRewriteReferences()`, reset→`pluginCleanup()`, seed→`pluginCopy()`, folder move→`SpecEntry` `target`, pre-copy→`SpecEntry`, inject→`pluginInjectSections()`), while remaining the parity reference (NFR-0001) for the original `plugin_generator.py`. There is no longer a competing as-is procedural architecture in the requirement set.
- **AC-8 — Override is a directive, not deferred.** The `overwrite` filename directive provides per-file override now, realized by `fileApplyOverrides()`. The earlier "bundle, don't override" framing is replaced by the processor model: default behavior is `fileBundle()` concatenation; `overwrite` prunes earlier entries via `fileApplyOverrides()`.
- **AC-10 — Path and content are two responsibilities.** `fileRename()` changes only the target path; `pluginRewriteReferences()` changes only content. They are never fused (unlike the original `copy_core_tree`). This is the maintainer's explicit requirement and the headline of the unification (FR-ARCH-0043/0049, FR-COPY-0032).
- **AC-11 — Every step is a processor at its tier; no out-of-band passes.** `reset`→`pluginCleanup()` and seeding→`pluginCopy()` (head `PluginProcessor`s); a folder move→`SpecEntry` `target`; `pre_copy_folders`→additional `SpecEntry`; `pre_move_files`→relocation `SpecEntry`/`fileRename()`; `pre_cleanup`/`post_cleanup`→not emitting (`target_contents=null`). Every step is a `FileProcessor` or `PluginProcessor` in a declared pipeline (FR-ARCH-0035).
- **AC-12 — Exact matching is generator-wide; references are observed not recomputed.** Every regular expression and match is anchored and boundary-delimited to the complete token (FR-ARCH-0037). `pluginRewriteReferences()` builds its lookup by reading the per-file `frames` (source VFS path → final target path) plus the `SpecEntry` folder pairs — the renames are observed from the frames, never re-derived from the rename rules (FR-ARCH-0049).
- **AC-13 — Generated content is built against final paths.** `pluginGenerateIndexes()`/`pluginInjectSections()` and any derived artifact are produced from post-`fileRename()` paths (FR-ARCH-0038), so generated content needs no reference rewriting; `pluginRewriteReferences()` touches only hand-authored body references. This minimizes both the work and the blast radius of reference rewriting.
- **AC-9 — RESOLVED. Directive delimiters are tilde fences + comma (`name.~order,tok,tok~.ext`).** Brackets `[ ]` were rejected: they are **git-pathspec magic** (`git add 'foo.[1a].md'` matches a glob, not the literal file), glob character-class metacharacters, and regex character-class openers. The chosen delimiters — `~` fences and `,` separators — are valid filenames on NTFS/APFS/ext4, **literal in git pathspecs**, **not** glob/extglob metacharacters (so they are safe both as data matched by patterns and unmatched as patterns), and inert in regex; `~` does not trigger shell tilde-expansion mid-filename (it is not at word start). The directive segment is also stripped at VFS-build (FR-ARCH-0020), so it never reaches a rename/reference regex. Grammar updated in FR-ARCH-0020/0021 and GLOSSARY. (`( )` were considered and are git/OS/glob-safe but carry shell-quoting and regex-escape caveats, so the fully-inert tilde/comma set was preferred.)

## Open Questions (OQ)

- **OQ-1 — RESOLVED.** Bundling is plain concatenation by `fileBundle()` with no markup/delimiters, in VFS order (set by order tokens / filename sort). No `<rosetta:file>`-style tags (unlike the server Bundler). See FR-ARCH-0042.
- **OQ-2 — RESOLVED.** Per-file override exists now via the `overwrite` directive (AC-8, FR-ARCH-0024/0041).
- **OQ-3 — RESOLVED.** Model normalization applies uniformly to all domains. Omission ("do not ship in any plugin") is a `SpecEntry.exclude` field — an array of VFS paths the entry must not emit (FR-ARCH-0002/0054, FR-COPY-0011). The legacy `rules/bootstrap.md` and `rules/local-files-mode.md` are listed there. This composes with `--domain` overlays (each entry/domain owns its omissions) and requires **no** source rename or directive — the source files stay unchanged because MCP serves them and instructions reference them.
- **OQ-4 — RESOLVED (subsumed).** Byte-identity now applies to **all** equivalent output, not just hook JSON: the parity bar is byte-for-byte everywhere (NFR-0001, AC-2). Hook JSON is therefore byte-identical as one case of the general rule.
- **OQ-5 — RESOLVED.** The VFS is built from structure + filename directives only (FR-ARCH-0011); each file's `frontmatter` slot is populated by `fileRead()` during processing (FR-ARCH-0040), not at VFS-build time, consistent with content I/O being confined to `fileRead`/`pluginWrite` (FR-ARCH-0033). The post-`fileRead()` working state carries the populated frontmatter.
